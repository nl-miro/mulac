# ADR-003: UUID v7 for All Generated Identifiers

**Status:** Accepted  
**Date:** 2026-05-12

## Context

The system generates identifiers in several places: message IDs when the transport does not supply one, reservation IDs when claiming an entry, and any future correlation tracking. Identifier choice affects:

- **Database index efficiency**: random (v4) UUIDs cause B-tree fragmentation and random I/O on inserts into large tables.
- **Sortability**: some operational queries (e.g. "show me reservations in order of acquisition") benefit from IDs that carry temporal information.
- **Uniqueness**: all identifiers must be globally unique without coordination.

UUID v4 satisfies uniqueness but not sortability or index locality. Sequential integers are sortable and index-friendly but require a coordination point (sequence generator) that does not exist in distributed workers.

## Decision

All identifiers generated by this system use UUID v7.

UUID v7 encodes a Unix millisecond timestamp in the high bits, followed by a random suffix. This means:
- IDs sort chronologically by default.
- B-tree index pages fill left-to-right, avoiding the fragmentation caused by v4.
- No coordination is required between workers — the random suffix ensures uniqueness even at millisecond collisions.

The `uuid` crate with the `v7` feature is the canonical source. Generation follows the pattern:

```rust
use uuid::Uuid;

let id = Uuid::now_v7();
```

When an external transport supplies a message ID (e.g. via AMQP `message_id` property), that value is used as-is after parsing; UUID v7 generation applies only when no ID is supplied.

## Consequences

**Benefits:**
- Monotonically increasing IDs improve PostgreSQL insert performance for UUID-keyed tables.
- IDs are inspectable — the leading bytes encode the creation time.
- Workers remain stateless and coordination-free.

**Trade-offs:**
- UUID v7 is not yet part of RFC 9562 (the current standard), though it is widely adopted and the `uuid` crate treats it as stable.
- The millisecond resolution of the timestamp means IDs generated in the same millisecond are ordered randomly within that window.
